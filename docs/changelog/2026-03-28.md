## 2. Team-fit Intake Became An Interview-First Profile Flow

### What Changed

- Replaced the older chip-heavy team-fit intake with a smaller profile shape:
  - one problem sentence
  - MBTI 5 axes
  - exactly 4 SDGs
  - one 800-character Markdown prompt
- Changed save behavior so the profile is only finalized after AI asks three sequential follow-up questions.
- Added follow-up-question support so saved profiles can keep growing one more question at a time.
- Simplified the Step 1 / Step 2 UI heavily:
  - quieter headings
  - cleaner labels/placeholders
  - centered SDG picker modal
  - lighter MBTI emphasis
  - an 80-character problem-sentence limit
  - a resettable 800-character prompt scaffold
- Added local draft persistence so guest-entered Step 1 / Step 2 content survives until a real saved profile exists.
- Added locale cache-busting so updated translation JSON is not silently hidden behind stale browser cache.

### Why It Mattered

- The product now collects richer motivation and collaboration context before recommending anyone.
- The saved profile is grounded in actual interview turns rather than only shallow form tags.
- The intake flow is lighter to read, more intentional to fill out, and more resilient during login or browser refresh transitions.

## 3. Recommendations Were Rebuilt Around Conversation Priority

### What Changed

- Moved recommendation logic from legacy `teamfit_profiles` assumptions toward `teamfit_explorer_profiles + interview turns`.
- Added extracted signal storage and recommendation embeddings to explorer profiles, refreshed on both initial save and follow-up answers.
- Replaced the old recommendation response with `safe_fit`, `complementary_fit`, and `wildcard_fit`.
- Scored candidates with interpretable factors:
  - problem resonance
  - role complementarity
  - work-style compatibility
  - value / SDG alignment
  - conversation potential
- Relaxed fallback slot-filling so recommendations can still return three usable candidates when the pool is small or problem phrasing is unusually different.
- Rewrote the “recommendation algorithm” explainer modal around the actual implemented logic so technical readers can understand the system at a glance.

### Why It Mattered

- Recommendations now match the product promise: “who should I talk to first?” rather than “who is my perfect teammate?”
- The backend now explains recommendation choices through readable factors instead of opaque single-score ranking.
- The recommendation engine is finally connected to the richer interview-first data model.

## 4. Saved Profiles Became A Real Workspace For Recommendation And Review

### What Changed

- Replaced the one-way saved-result surface with a tabbed workspace:
  - `추천 후보`
  - `팀핏 탐색 프로필`
- Made the saved profile editable again for Step 1 / Step 2 while keeping interview history as stored transcript data.
- Simplified interview transcripts to compact `Q : ... / A : ...` cards on both the root page and the candidate interview-history dialog.
- Added a detailed candidate modal used by both recommended people and general signed-up users.
- Added a signed-up-user directory under the main three recommendations:
  - paginated in groups of 10
  - filterable by fit-record presence
  - sortable by team-fit order or recency
- Added `teamfit_fit_checks` so a user can record `0~100` fit after a real conversation, from either a recommended candidate or someone discovered elsewhere.
- Excluded already-scored candidates from future recommendation results.
- Flattened and simplified the root UI so the workspace, recommendation cards, and directory cards feel like one coherent surface instead of several nested card layers.

### Why It Mattered

- Users can now move from profile writing to candidate review, to real conversation, to feedback capture in one continuous flow.
- The app no longer ends at “here are three people”; it now supports real-world narrowing and post-conversation learning.
- The UI is much lighter and easier to scan during demos, while still keeping rich detail behind click-through dialogs.
